# 검색 (Search) API - 쇼핑 검색 (Shopping Search) 레퍼런스

네이버 검색 결과를 뉴스, 백과사전, 블로그, 쇼핑, 웹 문서, 전문정보, 지식iN, 책, 카페글 등 각각의 다양한 도메인과 분야별로 볼 수 있는 폭넓은 API 생태계 가이드입니다. 오타 변환, 지역 검색 결과, 성인 검색어 판별 등의 다양한 부가 기능도 전천후로 제공하고 있습니다.
그 중에서 **쇼핑 검색 API**는 RESTful 형태로 요청을 구성해 네이버 검색 엔진의 데이터베이스에 존재하는 쇼핑몰 상품 검색 결과를 반환해 주는 매우 실용적인 API입니다. 결과 객체의 형태는 XML 포맷과 JSON 포맷 모두 다방면으로 유연하게 지원하고 있습니다.

## 1. 쇼핑 검색 개요 및 사전 공지 사항

### 1.1 검색 API 특징과 제약 사항
쇼핑 검색은 공용 네이버 검색 API의 뼈대를 그대로 재사용합니다. 즉, 호출 횟수 한도는 검색 API의 하루 정책을 따릅니다.
- **일일 API 호출 한도**: 하루 최대 25,000회 기본 제공.
- **방식**: 비로그인 방식의 오픈 API입니다.
- **의미**: 네이버 로그인을 통한 사용자 동의 창이나 OAuth에 관련된 Access Token(접근 토큰) 발급 등의 복잡한 처리 단계를 일체 생략하고, HTTP 통신 헤더(Header)에 `Client Id`와 `Client Secret` 정보만 단순 전송하여 즉시 데이터를 받아올 수 있는 특징을 지닙니다. 이로 인해 소상공인 웹페이지나 상용 솔루션에서도 손쉽게 플러그인처럼 내장할 수 있습니다.

### 1.2 서비스 연동 전 사전 준비 사항
쇼핑 검색 API를 상용망에서 사용하려면 다음과 같은 절차가 필수적으로 선행되어야 합니다.
1. [네이버 개발자 센터 (NAVER Developers)](https://developers.naver.com) 홈페이지에 방문합니다.
2. 애플리케이션 등록 대시보드를 통하여 새로운 클라이언트 애플리케이션 생성을 승인 받습니다.
3. 생성 시 출력되는 **클라이언트 아이디(Client ID)**, **클라이언트 시크릿(Client Secret)** 암호화 키를 복사해 안전한 위치에 보관합니다. 이 발급된 키들은 인증된 사용자 또는 검증된 앱 서버인지를 확인하는 가장 핵심적인 인증 수단이 됩니다.
4. 모든 비로그인 API 호출 시, 서버 쪽 로직 내부 HTTP 헤더에 이 두 값들이 명시적으로 주입되어야 합니다. 그렇지 않으면 API 프록시 서버에서 차단됩니다.
※ **주의 사항**: 애플리케이션을 최초 등록한 네이버 아이디는 관리자 권한을 영구히 소유하게 되므로 보안 관리에 각별히 유의해야 하며, 다수의 사내 직원이 운영하는 경우에는 '네이버 단체 회원 아이디'로 가입하여 운용할 것을 적극 권장합니다.

## 2. 쇼핑 검색 결과 조회 레퍼런스

### 2.1 기본 명세 및 호출 정의

쇼핑 검색 결과를 XML 또는 JSON 형태로 받기 위한 URL 형식과 데이터 전송 규칙입니다.

- **요청 프로토콜**: `HTTPS`
- **HTTP 메서드 (방식)**: `GET`

요청 URL은 응답받을 데이터의 포맷 유형에 따라 아래의 두 경로 중 하나로 호출해야 합니다.
- **XML 포맷 응답 요망 시**: `https://openapi.naver.com/v1/search/shop.xml`
- **JSON 포맷 응답 요망 시**: `https://openapi.naver.com/v1/search/shop.json`

### 2.2 요청 파라미터 (Query String Parameters)

이 API는 쿼리 스트링(Query String)으로 URL 뒤에 매개변수를 담아서 통신합니다.

| 분류 | 변수명 | 형 변환 | 필수 여부 | 설명 및 기입 규칙 |
| :--- | :--- | :--- | :---: | :--- |
| **질의어** | `query` | string | **Y** | 검색 엔진 서버로 보낼 구체적인 검색 키워드입니다. UTF-8 포맷으로 안전하게 URL 인코딩 처리를 거쳐서 삽입해야 합니다. |
| **결과 수** | `display` | integer | N | 한 번의 요청 페이지에 출력될 응답 아이템들의 개수를 지정합니다. 기본값은 10이며, 최대 100개까지 설정이 허용됩니다. 값이 한도를 돌파하면 오류가 납니다. |
| **시작점** | `start` | integer | N | 검색 결과 문서의 시작점 번호를 의미합니다. 기본 설정값은 1이며, 최대 1,000까지 셋팅 가능하여 페이지네이션(Pagination)의 기준값이 됩니다. |
| **정렬 옵션** | `sort` | string | N | 어떠한 기준에 맞추어 출력 리스트를 정렬할지 옵션을 고릅니다. <br>- `sim` : 정확도순으로 내림차순 정렬 (초기 기본값 설정)<br>- `date` : 날짜순으로 내림차순 정렬 (최신 쇼핑몰 등록일 순)<br>- `asc` : 가격오름차순 (저렴한 순 최우선)<br>- `dsc` : 가격내림차순 (고가 순 최우선, 고급 필터 옵션) |
| **제외 키워드** | `exclude` | string | N | 검색 결과에서 제외할 특정 상점 옵션이나 필터 세트를 선별할 수 있습니다. (예: `exclude=used:cbshop` 중고, 해외구매대행상품 배제 필터 도입 시 활용) 여러 파라미터 제외 시 `:` 로 연결합니다. |

### 2.3 HTTP 요청 헤더 (Headers)
API 요청에 대한 인증 권한을 증명하기 위해 GET 파라미터 외에도 다음 헤더를 필연적으로 포함해야 합니다.

```http
GET /v1/search/shop.xml?query=%EC%A3%BC%EC%8B%9D&display=10&start=1&sort=sim HTTP/1.1
Host: openapi.naver.com
User-Agent: curl/7.49.1
Accept: */*
X-Naver-Client-Id: {애플리케이션 등록 시 발급받은 클라이언트 아이디 값}
X-Naver-Client-Secret: {애플리케이션 등록 시 발급받은 클라이언트 시크릿 값}
```

## 3. 세부 응답 필드 명세 (Response Specification)

응답에 도달하고 상태 코드 `200`이 반환되면 지정된 포맷 기반으로 응답 스키마가 도출됩니다. JSON 문서 기준 스키마 정보입니다.

### 루트 요소 필드 (Root Elements)

| 필드 명칭 | 타입 | 설명 |
| :--- | :--- | :--- |
| `lastBuildDate` | string | 대상 검색 결과를 시스템 상에서 추출하여 마지막으로 제공한 일자를 포맷에 맞추어 반환 (`Tue, 04 Oct 2016 13:23:58 +0900`) |
| `total` | integer | 사용자가 조회한 쿼리에 대하여 데이터베이스에서 검색된 전체 종합 문서의 개수를 정수형 반환합니다. |
| `start` | integer | 상단 파라미터에서 요청한 문서의 검색 시작 지점 인덱스입니다. |
| `display` | integer | 현재 응답 결과 목록에서 반환된 검색 결과 문서의 실질 개수입니다. |
| `items` | array | **검색 쇼핑 결과의 상품 정보** 항목이 포함된 아이템 객체 배열입니다. 각 정보가 요소 안에 배치됩니다. |

### 요소 내부 개별 상품 필드 정보 (Items Array Elements)

아이템 배열 `items` 내부에 위치한 개별 상품 프로퍼티 값 표입니다.

| 상품 필드명 | 데이터 타입 | 설명 기재 사항 |
| :--- | :--- | :--- |
| `title` | string | 검색 결과가 포함된 상품의 이름입니다. 매칭된 단어는 `<b>` HTML 태그와 `</b>` 닫는 태그 사이로 기본 감싸져 하이라이팅 됩니다. |
| `link` | string | 상품을 제공하는 쇼핑몰 구매 페이지에 이르는 직통 URL. 혹은 네이버쇼핑 가격비교 화면 랜딩 페이지입니다. |
| `image` | string | 검색된 상품의 대표 섬네일 이미지에 사용되는 썸네일 이미지 파일의 원본 URL 링크. |
| `lprice` | integer | 네이버 쇼핑 플랫폼이 데이터베이스에서 파악한 해당 상품의 최저가. 기본적으로 정수형태로 등록됨. |
| `hprice` | integer | 수집된 최고가 데이터 (현재 신규 네이버 API 구조 로직상 제공되지 않아 0으로만 출력됨). |
| `mallName` | string | 상품을 입점시키고 실질적으로 판매하고 있는 고유 상점/쇼핑몰의 브랜드 명칭 (ex: "네이버 페이", "11번가" 등). |
| `productId` | integer | 네이버 쇼핑 엔진에서 발급하고 캐싱하고 있는 내부 관리 상품 ID 시리얼. 이 값을 기준으로 다른 파싱 작업이 가능함. |
| `productType` | integer | 파악된 상품의 종류 분류값. (일반상품, 서비스상품, 렌탈상품 등 다양한 상품군에 대한 번호 형태 값 부여 매칭 테이블). |
| `brand` | string | 상품의 소속 브랜드 이름 또는 상표명. 만약 해당 데이터가 존재하지 않는 경우 빈 문자열(`""`)로 반환됨. |
| `maker` | string | 실제 해당 상품을 조립 및 양산한 최상위 제조사 명칭 정보. 만일 매칭 데이터가 없으면 빈 문자열(`""`)이 송출됨. |
| `category1` | string | 쇼핑 카테고리 트리상 1차 최고 심도 (대분류) 명칭 제공. |
| `category2` | string | 쇼핑 카테고리 트리상 2차 심도 (중분류) 카테고리 명칭 반환. |
| `category3` | string | 쇼핑 카테고리 트리상 3차 심도 (소분류) 카테고리 분류 명칭 반환. |
| `category4` | string | 쇼핑 카테고리 트리상 4차 심도 분류 (세분류). 만약 데이터가 없으면 제공하지 않거나 빈 문자열로 넘어옴. |

## 4. 다양한 구현 예제 및 응답 예시 코드

### 4.1 리눅스 cURL 적용 예제 (XML 출력)
가장 간편한 서버 검증 방식인 cURL을 통과하여 데이터를 조회하는 방식입니다.
```bash
curl "https://openapi.naver.com/v1/search/shop.xml?query=%EC%A3%BC%EC%8B%9D&display=10&start=1&sort=sim" \
  -H "X-Naver-Client-Id: {애플리케이션 등록 시 발급받은 클라이언트 아이디 값}" \
  -H "X-Naver-Client-Secret: {애플리케이션 등록 시 발급받은 클라이언트 시크릿 값}" -v
```

## 5. 자주 마주치는 문제 및 오류 코드 가이드라인

API 연동 후 파싱 구간에서 문제가 생긴 경우, 네이버가 친절하게 HTTP 상태 코드와 함께 규격화된 에러 JSON 코드와 메시지를 송부합니다.

- **`SE01` (400 Bad Request)** : 파라미터가 유효하지 않은 경우입니다. `query` 파라미터가 누락되었거나 한글 검색어가 제대로 URL-Encode 처리(인코딩)되지 않았는지 언어의 인코더 펑션을 철저히 검사해 보세요.
- **`SE05` (400 Bad Request)** : API에서 받아들이는 허용 범위를 넘은 무효한 값을 삽입하려고 한 경우입니다. 주로 1,000 이상으로 제한된 `start` 파라미터 지점 범위를 넘겼거나, 100이 최고치인 `display` 파라미터 제한 값을 어겼는지 살펴봐야 합니다. 정렬 파라미터 `sort` 철자 오류도 주의하세요.
- **`SE09` (400 Bad Request)** : 올바르지 않은 형식의 파라미터를 보냈을 시. 문자열 값(`query`) 대신 특수 문자가 파싱을 오염시킨 경우.
- **`403 Forbidden`** : API를 사용할 최고 권한이 없습니다. 개발자 센터 API 탭에서 쇼핑 검색 기능이 V-Check 되어 승인처리되었는지 확실하게 다시 살펴보십시오.
- **`500 Internal Server Error`** : 네이버 서버군 오류이므로 재시도 처리 로직을 구상하여 작동시켜주시면 무난합니다.
